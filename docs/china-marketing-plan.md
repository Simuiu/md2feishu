# md2feishu 中国市场宣传方案

## 1. 这次要宣传什么

宣传对象不是单独的 Codex skill，而是：

> `md2feishu`：一个 CLI 工具 + Codex Skill，用来把 AI 编程工具生成的 Markdown 自动同步成飞书云文档。

一句话介绍：

> 让 Codex / Claude Code 写出的 Markdown，自动变成飞书云文档。

核心用户感知：

- AI 帮我写文档。
- `md2feishu` 帮我同步到飞书。
- Git 帮我保存历史版本。
- 飞书只展示最新版。

## 2. 目标用户

第一批重点打这 4 类人：

| 用户 | 痛点 | 适合讲法 |
| --- | --- | --- |
| Codex / Claude Code / Cursor 用户 | AI 输出 Markdown，但团队看飞书 | AI 文档最后一公里 |
| 产品经理 / 技术负责人 | PRD、方案、调研报告要反复同步 | 不用再手动导入飞书 |
| 独立开发者 / Vibe Coding 用户 | 想把自己的 AI 工作流工具化 | 一个真实痛点做成开源工具 |
| 求职面试受众 | 想展示 AI 编程落地能力 | 从痛点到开源项目的完整案例 |

## 3. 核心卖点

宣传时固定讲这 5 个点：

1. AI 工具很会写 Markdown，但 Markdown 不适合直接给团队协作阅读。
2. 团队常用飞书看文档，但每次手动导入很麻烦。
3. `md2feishu sync` 可以把本地 Markdown 同步成真正的飞书云文档，不是 `.md` 附件。
4. 同一个本地文件后续更新，会覆盖同一个飞书文档，飞书永远是最新版。
5. 历史版本不靠 V1/V2 文件，交给 Git 管理，还能对比差异。

## 4. 推荐宣传口号

主口号：

> 让 AI 写出的 Markdown，自动变成飞书云文档。

副口号：

> Markdown 留在本地，版本交给 Git，最新版交给飞书。

更口语化版本：

> 终于不用把 AI 写的文档反复导入飞书了。

## 5. 渠道策略

### 5.1 V2EX

目标：

- 获取第一批开发者反馈。
- 看有没有真实用户愿意 star、试用、提 issue。

内容形式：

- 发在“分享创造”。
- 语气要像真实开发者分享，不要像广告。

标题建议：

> 开源了一个小工具：把 Codex / Claude Code 生成的 Markdown 自动同步成飞书云文档

重点讲：

- 自己真实痛点。
- 为什么不是普通 Markdown 转换器。
- GitHub 地址。
- 邀请大家试用和提建议。

### 5.2 掘金

目标：

- 技术人群曝光。
- 让项目看起来更有技术含量。

内容形式：

- 技术长文。

标题建议：

> 我做了一个 Codex Skill，把 AI 生成的 Markdown 自动同步到飞书

文章结构：

1. 痛点：AI 写文档强，但交付到飞书麻烦。
2. 方案：Markdown + Git + Feishu。
3. 实现：CLI、lark-cli、Codex Skill。
4. 设计取舍：飞书展示最新版，Git 保存历史。
5. 开源地址和后续路线图。

### 5.3 小红书

目标：

- 触达 AI 工具用户、产品经理、知识工作者。
- 用更轻的方式讲痛点。

内容形式：

- 6 张图文卡片。

标题建议：

> 终于不用把 AI 写的文档反复导入飞书了

卡片结构：

1. 封面：痛点标题。
2. 以前：AI 写 Markdown，手动导入飞书。
3. 问题：每改一版都要重新导入，文档越来越乱。
4. 现在：`md2feishu sync` 自动同步。
5. 结果：飞书只看最新版，本地 Git 管历史。
6. 结尾：开源地址 + 适合 Codex / Claude Code 用户。

### 5.4 B 站 / 视频号

目标：

- 用录屏快速说明工具价值。
- 给 GitHub README 或求职作品集提供演示素材。

内容形式：

- 2 分钟演示视频。

视频结构：

1. 0-15 秒：展示痛点。
2. 15-45 秒：终端执行 `md2feishu sync docs/plan.md`。
3. 45-75 秒：飞书出现格式化云文档。
4. 75-105 秒：修改 Markdown 后再次同步，飞书还是同一个文档最新版。
5. 105-120 秒：GitHub 开源地址和适用人群。

标题建议：

> 我做了个工具，把 AI 写的 Markdown 自动同步到飞书

### 5.5 知乎 / 公众号

目标：

- 做长期内容沉淀。
- 服务搜索和求职展示。

内容形式：

- 知乎回答或公众号长文。

标题建议：

> 我如何把一个真实 AI 工作流痛点做成开源工具

重点讲：

- 这不是玩具 Demo。
- 是自己真实使用 Codex / Claude Code 时遇到的痛点。
- 从需求定义、规则沉淀、工具实现、测试、GitHub 发布到推广，形成完整闭环。

## 6. 七天发布节奏

| 日期 | 动作 | 目标 |
| --- | --- | --- |
| Day 1 | 完善 GitHub README、图片、Release、Social preview | 让仓库看起来可信 |
| Day 2 | 发 V2EX 首发帖 | 拿第一批开发者反馈 |
| Day 3 | 发掘金技术长文 | 建立技术可信度 |
| Day 4 | 发小红书图文 | 触达非纯技术用户 |
| Day 5 | 发 B 站 / 视频号演示视频 | 用视频降低理解成本 |
| Day 6 | 发知乎或公众号长文 | 沉淀完整项目故事 |
| Day 7 | 发复盘帖 | 总结反馈，继续引导 star 和 issue |

## 7. 首批要制作的营销物料

建议只先做这 8 个，不要一开始铺太散：

1. V2EX 首发帖。
2. 掘金技术长文。
3. 小红书 6 张图文卡片文案。
4. B 站 2 分钟视频脚本。
5. 公众号 / 知乎长文。
6. 朋友圈 / 微信群短文案。
7. GitHub README 中文简介优化。
8. 求职作品集版项目介绍。

## 8. 成功指标

第一阶段不要只看下载量，先看反馈质量：

| 指标 | 目标 |
| --- | --- |
| GitHub Star | 30-100 |
| V2EX 评论 | 10+ |
| 掘金收藏 | 20+ |
| 小红书收藏 / 私信 | 有人问怎么安装 |
| GitHub Issue | 有真实问题或功能建议 |
| 延伸需求 | 有人问是否支持语雀、Notion、企业微信文档 |

## 9. 需要你确认的问题

在开始制作营销物料前，需要确认：

1. 你希望主打“开发者工具”，还是主打“AI 文档工作流”？
2. 你是否愿意在内容里强调“这是我用 Codex 协作完成的开源工具”？
3. 第一波是否先只发 V2EX + 掘金，等反馈后再做小红书和视频？
4. GitHub Star 是否作为第一阶段核心目标？
5. 内容里是否要带“找工作 / 作品集案例”的表达，还是先不公开讲？

## 10. 我建议的执行顺序

如果你确认方案，我建议按这个顺序制作物料：

1. V2EX 首发帖。
2. 掘金技术长文。
3. 小红书图文卡片文案。
4. B 站视频脚本。
5. 求职作品集版项目介绍。

这样先验证开发者反馈，再扩展到更泛的人群。
